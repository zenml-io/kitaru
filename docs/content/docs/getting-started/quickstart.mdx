This guide walks you through installing Kitaru, verifying the CLI, and running
your first small flow.

Kitaru requires **Python 3.11 or newer**.

## Install

<Tabs items={["uv (recommended)", "pip"]}>
  <Tab value="uv (recommended)">
    ```bash
    uv add kitaru
    ```
  </Tab>
  <Tab value="pip">
    ```bash
    pip install kitaru
    ```
  </Tab>
</Tabs>

The base install is enough to run flows locally against the default local
database.

## Optional: add local server support

If you also want the local dashboard and REST API, install the local extra:

<Tabs items={["uv (recommended)", "pip"]}>
  <Tab value="uv (recommended)">
    ```bash
    uv add kitaru --extra local
    ```
  </Tab>
  <Tab value="pip">
    ```bash
    pip install "kitaru[local]"
    ```
  </Tab>
</Tabs>

## Verify the CLI

<Tabs items={["uv project", "pip environment"]}>
  <Tab value="uv project">
    ```bash
    uv run kitaru --version
    uv run kitaru --help
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    kitaru --version
    kitaru --help
    ```
  </Tab>
</Tabs>

## Optional: start a local Kitaru server

<Callout type="info">
  Flows run locally without any login. Bare `kitaru login` is optional: it
  starts a local server so you can browse executions in the dashboard and use
  the REST API while keeping the same local runtime underneath.
</Callout>

<Tabs items={["uv project", "pip environment"]}>
  <Tab value="uv project">
    ```bash
    uv run kitaru login
    uv run kitaru status
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    kitaru login
    kitaru status
    ```
  </Tab>
</Tabs>

<Callout type="info">
  Local server startup is currently supported on Unix-like systems and WSL. If
  you are on native Windows, use WSL for this flow.
</Callout>

## Optional: connect to an existing remote Kitaru server

If you already have a deployed Kitaru server and want this quickstart to use
it, connect first, verify with `kitaru status`, and then run the same flow
commands below.

<Tabs items={["uv project", "pip environment"]}>
  <Tab value="uv project">
    ```bash
    uv run kitaru login https://my-server.example.com --api-key <API_KEY>
    # add --project <PROJECT> or other remote-login flags if your setup requires them
    uv run kitaru status
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    kitaru login https://my-server.example.com --api-key <API_KEY>
    # add --project <PROJECT> or other remote-login flags if your setup requires them
    kitaru status
    ```
  </Tab>
</Tabs>

## Initialize your project

<Tabs items={["uv project", "pip environment"]}>
  <Tab value="uv project">
    ```bash
    uv run kitaru init
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    kitaru init
    ```
  </Tab>
</Tabs>

This creates a `.kitaru/` directory that marks your project root. Kitaru uses
it for source root detection when packaging flows for remote execution.

## Run your first flow

Create a small script named `demo.py`:

```python
from kitaru import checkpoint, flow

@checkpoint
def fetch(topic: str) -> str:
    return f"notes about {topic}"

@flow
def demo(topic: str) -> str:
    return fetch(topic).upper()

handle = demo.run("kitaru")
print("Execution:", handle.exec_id)
print("Result:", handle.wait())
```

Then run it:

<Tabs items={["uv (recommended)", "pip environment"]}>
  <Tab value="uv (recommended)">
    ```bash
    uv run demo.py
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    python demo.py
    ```
  </Tab>
</Tabs>

What happens here:

1. `@checkpoint` marks a durable unit of work
2. `@flow` defines the outer execution boundary
3. `demo.run(...)` starts a new execution and returns a `FlowHandle`
4. `handle.wait()` blocks until the execution finishes and returns the flow result

If you want to inspect that execution later, the execution ID you printed is the
same ID used by `KitaruClient` and `kitaru executions ...`.

## Prefer a repo example?

If you would rather run a maintained example from this repository than type out
your own `demo.py`, start here:

```bash
uv sync --extra local
uv run examples/basic_flow/first_working_flow.py
```

For the broader catalog, see [Examples](/getting-started/examples).
