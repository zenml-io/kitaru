This guide walks you through setting up a model and running your first durable
flow. If you haven't installed Kitaru yet, start with
[Installation](/getting-started/installation).

## Set up a model

`kitaru.llm()` needs to know which model to call and how to authenticate.
The quickest way is to register a model alias:

```bash
kitaru secrets set openai-creds --OPENAI_API_KEY=sk-...
kitaru model register fast --model openai/gpt-4o-mini --secret openai-creds
```

This stores your API key in a Kitaru secret and creates an alias called `fast`
that points to `openai/gpt-4o-mini`. The first alias you register becomes the
default, so `kitaru.llm()` calls will use it automatically.

<Callout type="info">
  You can skip registration and just set `OPENAI_API_KEY` in your environment —
  `kitaru.llm(model="openai/gpt-4o-mini")` will pick it up. Aliases are useful
  when you want to swap models or credentials without changing code. See
  [Secrets + Model Registration](/guides/secrets-and-model-registration) for
  the full setup.
</Callout>

<Tabs items={["uv project", "pip environment"]}>
  <Tab value="uv project">

  </Tab>

    ```bash

    ```
  </Tab>

This creates a `.kitaru/` directory that marks your project root. Kitaru uses
it for source root detection when packaging flows for remote execution.

## Run your first flow

Create a file called `agent.py`:

```python
import kitaru
from kitaru import checkpoint, flow

@checkpoint
def research(topic: str) -> str:
    return kitaru.llm(f"Summarize {topic} in two sentences.")

@checkpoint
def draft_report(summary: str) -> str:
    return kitaru.llm(
        f"Write a short report based on this summary:\n\n{summary}"
    )

@flow
def research_agent(topic: str) -> str:
    summary = research(topic)
    return draft_report(summary)

if __name__ == "__main__":
    research_agent.run("durable execution for AI agents")
```

Then run it:

<Tabs items={["uv (recommended)", "pip environment"]}>
  <Tab value="uv (recommended)">
    ```bash
    uv run agent.py
    ```
  </Tab>
  <Tab value="pip environment">
    ```bash
    python agent.py
    ```
  </Tab>
</Tabs>

What happens here:

1. `@flow` marks the top-level execution boundary — everything inside is tracked
2. Each `@checkpoint` persists its return value automatically
3. `kitaru.llm()` resolves your default alias, injects the linked secret, calls the model, and captures the prompt, response, token usage, and cost
4. `.run()` starts the execution and blocks until completion

## Why checkpoints matter

Now imagine `research` succeeds but `draft_report` fails — maybe you hit a rate
limit or the model returned an error. Without Kitaru you would re-run the entire
script, paying for the `research` call again and losing progress. With Kitaru, you replay from the
failure point:

```bash
kitaru executions replay <exec-id> --from draft_report
```

Replay creates a new execution that **reuses the recorded output of `research`**
and only re-executes `draft_report`. The more checkpoints your flow has, the
less work you repeat when something goes wrong.
