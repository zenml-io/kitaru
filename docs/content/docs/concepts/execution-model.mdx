This page explains how Kitaru executes your workflows and what to expect when
things succeed or fail.

## The mental model

Kitaru tracks your workflow at **flow and checkpoint boundaries**. Think of it
like this:

1. A [flow](/concepts/flows) defines the outer durable boundary
2. [Checkpoints](/concepts/checkpoints) inside the flow are the persisted work
   units
3. Kitaru records the outcome of each checkpoint as it runs
4. On re-execution, completed checkpoints can return persisted results instead of
   running again

Your Python code between checkpoints runs normally — Kitaru does not intercept
every line. The durability guarantees apply at the checkpoint boundaries.

## What happens when you run a flow

```python
handle = my_agent.run(url="https://example.com")
# execution runs in the background
result = handle.wait()
```

Behind the scenes:

<Steps>
  <Step>
    Kitaru submits a new execution
  </Step>
  <Step>
    `.run()` returns a `FlowHandle` immediately
  </Step>
  <Step>
    The execution runs in the background
  </Step>
  <Step>
    Calling `handle.wait()` blocks until the execution finishes
  </Step>
  <Step>
    On success, the flow's return value is extracted and returned
  </Step>
  <Step>
    On failure, `handle.wait()` raises `KitaruExecutionError` (or a more
    specific subclass)
  </Step>
</Steps>

For a synchronous one-liner, chain `.wait()`:

```python
result = my_agent.run(url="https://example.com").wait()
```

## How errors surface

If a flow execution fails, `handle.wait()` raises a typed
`KitaruExecutionError`:

```python
import kitaru

try:
    result = my_agent.run(topic="bad-input").wait()
except kitaru.KitaruExecutionError as exc:
    print(exc.exec_id, exc.status, exc.failure_origin)
```

## Result reuse

By default, checkpoint result reuse is **enabled** (`cache=True` on `@flow`).
This means:

- Checkpoints that already succeeded in a previous execution can return their
  recorded output without re-running
- This is especially useful during development when you want to iterate on later
  checkpoints without repeating expensive earlier work

You can disable result reuse at the flow level or per-run:

```python
from kitaru import flow

# Disable result reuse by default
@flow(cache=False)
def always_fresh(url: str) -> str:
    ...

# Or override per-run
handle = my_agent.run("https://example.com", cache=False)
```

<Callout type="info">
  **Dapr backend:** The `cache` setting is accepted but currently ignored.
  The Dapr workflow engine does not provide cross-execution result reuse —
  checkpoints always re-execute on each run. Within a single execution, the
  deterministic workflow interpreter ensures completed checkpoints return
  cached results on replay.
</Callout>
