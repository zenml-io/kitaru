A **flow** is the outer durable boundary in Kitaru. It marks the top-level
function whose execution becomes persistent, replayable, and observable.

## Defining a flow

Decorate your orchestration function with `@flow`:

```python
from kitaru import flow

@flow
def my_agent(url: str) -> str:
    data = fetch_data(url)
    return process_data(data)
```

The decorated function becomes a callable wrapper object. Inside the flow body,
you compose [checkpoints](/concepts/checkpoints) — the units of work whose
outputs are persisted.

## Two ways to run a flow

Every `@flow` supports two invocation styles. They both run the same underlying
durable execution — they differ only in how you interact with the result.

### `.run()` (the standard way)

```python
handle = my_agent.run("https://example.com")
print(handle.exec_id)   # unique execution identifier
# ... do other work ...
result = handle.wait()   # block until finished
```

`.run()` submits the execution and immediately returns a `FlowHandle`. The flow
runs in the background while your code continues. Call `handle.wait()` when you
need the result.

For a synchronous one-liner, chain `.wait()`:

```python
result = my_agent.run("https://example.com").wait()
```

### `.deploy()` (remote intent)

```python
handle = my_agent.deploy("https://example.com", stack="production")
```

`.deploy()` is semantic sugar for `.run()` that signals remote execution
intent. Today it behaves identically to `.run()` with a `stack` parameter —
it exists to make your code more readable when the intent is deployment rather
than local development.

## FlowHandle

A `FlowHandle` is returned by `.run()` and `.deploy()`. It gives you
access to the running execution:

| Property / Method | What it does |
|---|---|
| `handle.exec_id` | The unique execution identifier (a string you can store or log) |
| `handle.status` | Current execution status (refreshed on each access) |
| `handle.wait()` | Block until the execution finishes, then return the result |
| `handle.get()` | Return the result immediately if finished, otherwise raise an error |

<Callout type="info">
  `handle.get()` does **not** wait. If the execution is still running, it raises
  a `RuntimeError`. Use `handle.wait()` when you want to block.
</Callout>

## Runtime options

You can configure execution behavior at the decorator level (defaults) or
override per-run via `.run()` and `.deploy()`:

```python
from kitaru import flow

@flow(retries=2, cache=False)
def my_agent(url: str) -> str:
    ...

# Override at call time
handle = my_agent.run("https://example.com", retries=3, cache=True)
```

| Option | Default | What it controls |
|---|---|---|
| `retries` | `0` | Number of automatic retries on failure |
| `cache` | `True` | Whether checkpoint results can be reused from previous runs |
| `stack` | `None` | Target execution environment for this run (overrides the active stack default) |
| `image` | `None` | Container image for remote execution (string, mapping, or settings object). Supports `base_image`, `requirements`, `environment`, `apt_packages`, `replicate_local_python_environment`, and `dockerfile`. |

Per-run values override decorator defaults. If you don't pass an override, the
decorator default applies.

When neither per-run nor decorator `stack` values are set, Kitaru uses your
active stack. You can inspect or change it with `kitaru stack current` and
`kitaru stack use <name-or-id>` (or the matching SDK helpers).

## Rules to know

- **Flows cannot nest.** You cannot call a `@flow` from inside another flow.
- **Flow functions should compose checkpoints.** The flow body is the
  orchestration layer — heavy work belongs in
  [checkpoints](/concepts/checkpoints).
- **Use `.run()` to invoke flows.** Direct calls (`my_agent(...)`) are not
  supported. Use `my_agent.run(...)` to start an execution.
- **Retries must be non-negative.** Passing a negative `retries` value raises a
  `ValueError`.
